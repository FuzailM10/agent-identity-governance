Agent Identity Governance

A governance control plane for AI agents —
register, authorize, audit & revoke them like human employees.

Muhammad Fuzail · FuzailM10

Live: agent-identity-governance.vercel.app

The problem

  • AI agents act in the cloud — approving, paying, querying, deploying.
  • But they act under a human's credentials.
  • When something breaks: was it the employee, or the executive whose account it used?

Human identity is solved. Non-human identity (NHI) — service accounts, API keys, AI agents — is not.

The thesis

An AI agent should:

  • have its own governed identity,
  • act only within explicitly granted, time-boxed permissions,
  • leave a tamper-evident trail that always points back to an accountable human.

A layer, not a new IAM platform

  • Enterprises already run Okta, Entra, AWS IAM. Nobody replaces those.
  • This is the missing governance layer on top — it brokers what agents may do.
  • It federates with existing IAM; it isn't the identity provider.

Architecture

  Dashboard (web UI)  ── the face: see & drive it
        │  HTTPS / JSON
  Control Plane (FastAPI) ── the real logic
    identity · tokens · policy
    audit · risk · kill switch
        │
  Postgres (Neon) ── identity + tamper-evident audit

Provider-agnostic core → new environments plug in as adapters.

① Identity

  • Owner (human) ⟶ Agent (non-human), permanently linked.
  • No orphan agents — every agent has an accountable human.
  • Each agent issued a SPIFFE-style ID spiffe://aig/agent/… (workload-identity standard).

/attribution answers: “which human is behind this agent?”

② Just-in-time capability tokens

  • A grant = scope (invoice:approve) + constraints (≤ $5k) + TTL.
  • Issued as a signed JWT — tamper-evident, self-describing.
  • Carries owner_id → attribution travels inside the access.
  • Auto-expires → no standing privilege (least privilege).

③ Policy broker

Every action is brokered → one of:

  • ALLOW — within scope & limits
  • STEP-UP — over a limit → a human must approve
  • DENY — out of scope / invalid token

Single choke point. Rules move to OPA/Rego in production. Finer-grained than RBAC (context-aware → ABAC/PBAC).

④ Tamper-evident audit

  • Every decision logged in a hash chain (SHA-256).
  • Each entry hashes its contents + the previous hash.
  • Alter any past record → every later hash breaks → detected.

Even direct DB access can't silently rewrite history. /audit/verify proves it.

⑤ Adaptive risk

  • Each agent has a live risk score (0–100).
  • Denied attempts push it up; good behavior cools it down.
  • Past a threshold → policy auto-tightens (ALLOW ⟶ STEP-UP).

Risk-based / adaptive access control — the system tightens around a misbehaving agent by itself.

⑥ Kill switch

  • One call ⟶ agent killed + all grants revoked.
  • Next action DENIED instantly — even on an unexpired token.
  • The kill is itself written to the audit trail.

Stack & deployment

  • FastAPI control plane · SQLAlchemy · PyJWT
  • Postgres (Neon, cloud) / SQLite (local) — one-line swap
  • Vercel serverless + GitHub — auto-deploys on every push

Production hardening (chosen scope)

  • Policy → OPA/Rego; tokens → asymmetric/rotating keys
  • Owner login → Keycloak / OIDC; identity → SPIRE
  • Externally notarize the audit head; authn/z + rate-limit the control plane

Named on purpose — the edges were chosen, not missed.
